# 第八章：内置消息类型

本章把第五章的基础类型扩展为可用于三维可视化的复合消息类型。
教学目标不是追求官方 SDK 的 protobuf 完整实现，而是用当前项目一致的 JSON 路径把消息建模、序列化、测试、示例串起来。

---

## 1. 为什么要有“内置消息类型”

在 Foxglove 的使用场景里，常见数据不是单个标量，而是“可视化对象”。
比如：

1. 一个立方体（位置、姿态、尺寸、颜色）
2. 一条线（点集、索引、粗细、颜色）
3. 一个实体集合的增删改（SceneUpdate）
4. 坐标系变换（FrameTransform）
5. 日志（Log）
6. 压缩图像（CompressedImage）

如果只靠 `Vector3`、`Color`、`Pose`，上层业务会自己拼结构，容易字段不统一。
内置类型的意义是“约定字段、稳定语义、可直接消费”。

## 3. 数据建模原则：组合优先

这些类型全部由前几章已有基础类型组合而来：

- 姿态用 `Pose`
- 坐标用 `Vector3` 或 `Point3`
- 颜色用 `Color`
- 时间戳用 `Timestamp`
- 生存期用 `Duration`

这样做的好处：

1. 序列化规则复用（NaN/null 等）
2. Schema 结构可读
3. 单元测试可以逐层验证
4. 类型边界清晰，后续扩展更稳

---

## 4. 字段顺序与 JSON 稳定性

本项目要求 JSON 字段按字母序输出。
这不是“美观问题”，而是为了：

1. golden 文件可精确比较
2. 输出确定性（deterministic output）
3. 避免不同平台/实现下顺序抖动

示例（`FrameTransform`）：

```cpp
void to_json(nlohmann::json& j, const FrameTransform& transform) {
  j = nlohmann::json{
    {"child_frame_id", transform.child_frame_id},
    {"parent_frame_id", transform.parent_frame_id},
    {"rotation", transform.rotation},
    {"timestamp", transform.timestamp},
    {"translation", transform.translation}
  };
}
```

## 5. 可选字段策略（缺省不覆盖）

`from_json` 对复杂类型采用“字段存在才读取”的策略：

- JSON 缺字段：保留结构体默认值
- JSON 有字段：覆盖默认值

这是复合消息最实用的反序列化策略，适合渐进演进与部分字段传输。

```cpp
void from_json(const nlohmann::json& j, SceneEntity& entity) {
  if (j.contains("frame_id")) {
    j.at("frame_id").get_to(entity.frame_id);
  }
  if (j.contains("frame_locked")) {
    j.at("frame_locked").get_to(entity.frame_locked);
  }
}
```

## 6. 枚举序列化：统一走整数

三个枚举都按整数编码：

- `LineType`：0/1/2
- `DeletionType`：0/1
- `LogLevel`：0~5

优点是：

1. 与 schema 中 integer 对齐
2. 网络载荷更紧凑
3. 与官方 schema 风格一致

```cpp
{"type", static_cast<uint8_t>(primitive.type)}
```

## 7. CompressedImage 的 base64 处理

`CompressedImage::data` 是二进制字节数组，JSON 里必须变成字符串。
本章在 `src/messages.cpp` 提供了本地 base64 编码/解码函数：

- `encode_base64(const std::vector<uint8_t>&)`
- `decode_base64(const std::string&)`

序列化：`vector<uint8_t> -> base64 string`
反序列化：`base64 string -> vector<uint8_t>`

示例输出：

```json
{"data":"/9j/2wABAg==","format":"jpeg","frame_id":"camera","timestamp":{"nsec":10,"sec":9}}
```

`SceneEntity` 是“一个可视化实体”，同时可带多种 primitive：

- `cubes`

并带有：

- `entities`：新增或更新

这对应可视化系统里最常见的“增量更新”模式。

## 9. TypedChannel：在 RawChannel 之上加类型约束

`RawChannel` 接收字节流。
`TypedChannel<T>` 把 `encode(T)` 与 `RawChannel::log()` 绑定，降低调用方出错概率。

```cpp
template <typename T>
class TypedChannel {
public:
  explicit TypedChannel(RawChannel channel) : channel_(std::move(channel)) {}

  FoxgloveResult<void> log(const T& msg, uint64_t timestamp_ns) {
    auto json = encode(msg);
    if (!json.has_value()) {
      return tl::make_unexpected(json.error());
    }
    const auto& data = json.value();
    channel_.log(reinterpret_cast<const uint8_t*>(data.data()), data.size(), timestamp_ns);
    return {};
  }
};
```

针对高频类型再给出别名：

- `SceneUpdateChannel`
- `FrameTransformChannel`
- `LogChannel`

## 10. JSON Schema 输出策略

每个新增类型都实现 `json_schema()`，保持与前面章节一致：

1. `type: object`
2. `properties` 完整列出
3. `required` 对齐字段定义
4. 枚举字段使用 integer + 范围约束

这让 schema 不只是注释，而是可用于校验、文档、调试。

## 11. Golden 文件测试策略

本章新增 11 个 golden 文件：

- `cube_primitive.json`
- `sphere_primitive.json`
- `cylinder_primitive.json`
- `arrow_primitive.json`
- `line_primitive.json`
- `scene_entity_deletion.json`
- `scene_entity.json`
- `scene_update.json`
- `frame_transform.json`
- `log.json`
- `compressed_image.json`

格式规则：

1. 单行紧凑 JSON
2. 不带结尾换行
3. 字段顺序固定

测试层面做两件事：

1. roundtrip（encode->decode->==）
2. golden match（字符串精确相等）

## 14. 常见坑与排查建议

1. **golden mismatch**
   - 先检查字段顺序
   - 再检查 double 精度与是否 NaN/null
2. **decode 失败**
   - 看是否缺少必需字段
   - 看枚举值是否越界
3. **CompressedImage 失败**
   - 检查 base64 padding
   - 检查非法字符
4. **roundtrip 不相等**
   - 优先看默认值是否被意外覆盖
