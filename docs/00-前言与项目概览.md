## 1. Foxglove生态系统简介

### 1.1 什么是Foxglove

[Foxglove](https://foxglove.dev) 是一个面向机器人多模态数据的可观测性与可视化平台。在现代机器人系统中，我们需要同时处理传感器数据（激光雷达、摄像头、IMU）、控制指令、算法中间结果等多种数据流。Foxglove提供了统一的解决方案：

- **Foxglove Studio**: 基于Web的实时可视化工具，支持3D场景渲染、图表绘制、日志回放
- **MCAP格式**: 专为机器人数据设计的高性能日志存储格式
- **WebSocket协议**: 用于实时数据流传输的通信协议

### 1.2 Foxglove SDK架构

官方Foxglove SDK采用分层架构设计：

```
┌─────────────────────────────────────────┐
│  Application Layer (Your Code)          │
├─────────────────────────────────────────┤
│  C++ SDK (foxglove::*)                  │  ← 面向用户的C++接口
├─────────────────────────────────────────┤
│  C FFI Layer (foxglove_c_*)             │  ← C语言绑定层
├─────────────────────────────────────────┤
│  Rust Core (foxglove crate)             │  ← 核心实现（Rust）
├─────────────────────────────────────────┤
│  Runtime (async I/O, WebSocket, MCAP)   │  ← 异步运行时
└─────────────────────────────────────────┘
```

这种分层设计有其合理性：Rust提供内存安全和性能，C层提供语言互操作性，C++层提供面向对象的便利接口。然而，对于学习目的而言，这种架构增加了认知负担——你需要同时理解三种语言的交互。

## 2. 本教程构建什么

### 2.1 项目目标

本教程将从头构建一个**纯C++17**实现的Foxglove SDK核心子集。我们的目标是：

1. **教育性**: 每一行代码都解释"为什么这样设计"
2. **完整性**: 涵盖Channel、Schema、WebSocket服务器、MCAP写入等核心概念
3. **实用性**: 最终能够连接Foxglove Studio进行实时可视化

### 2.2 与官方SDK的关系

| 维度 | 官方SDK | 本教程实现 |
|------|---------|-----------|
| 实现语言 | Rust + C FFI + C++ | 纯C++17 |
| 功能范围 | 完整功能集 | 核心子集 |
| 目标用户 | 生产环境开发者 | 学习者、研究者 |
| 代码复杂度 | 高（跨语言抽象） | 中等（单一语言） |
| 依赖数量 | 较多（Rust生态） | 精简（C++生态） |

选择纯C++实现并非否定官方设计，而是为了教学清晰性。当你理解纯C++实现后，再阅读官方SDK源码会有更深的理解。

## 6. 设计理念

### 6.1 错误处理策略

我们采用C++17的异常安全设计原则，结合 `tl::expected` 实现显式错误传播：

```cpp
// 使用 expected 替代异常作为主要错误传播机制
tl::expected<Channel, Error> create_channel(const std::string& topic);

// 调用方必须处理错误
auto result = create_channel("/camera/image");
if (!result) {
    log_error(result.error());
    return;
}
auto channel = std::move(result.value());
```

### 6.2 内存管理

遵循RAII原则，避免裸指针：

```cpp
// Good: 使用智能指针和值语义
class WebSocketServer {
    std::unique_ptr<lws_context, lws_context_destroyer> context_;
    std::vector<std::shared_ptr<Client>> clients_;
};

// Bad: 裸指针需要手动管理
lws_context* context_;  // 谁负责delete?
```

### 6.3 并发模型

WebSocket服务器采用单线程事件循环模型：

```cpp
// libwebsockets基于事件回调
// 所有操作在单个线程中顺序执行
// 避免多线程数据竞争
```

对于生产者线程，我们使用无锁队列或线程安全的消息队列进行解耦。
