# 第五章：消息序列化

本章实现 `Timestamp`、`Duration`、`Vector3`、`Quaternion`、`Pose`、`Color` 等基础消息类型的 JSON 序列化。与官方 SDK 使用 protobuf 不同，我们采用纯 JSON 序列化作为教学简化，展示 ADL（Argument Dependent Lookup）模式的设计与应用。

## 核心概念

### JSON 序列化策略

Foxglove 协议支持多种消息编码格式，其中 JSON 是最直观、最易调试的格式。我们选择 JSON 而非 protobuf 作为本章的教学重点，原因有三：

1. **可见性**：JSON 是纯文本格式，便于观察和理解序列化过程
2. **调试友好**：人类可读的格式便于排查问题
3. **设计模式清晰**：ADL 模式在 JSON 场景下更易于讲解

```cpp
// 序列化：C++ 结构体 → JSON 字符串
Timestamp ts{1234567890, 123456789};
auto json_str = foxglove::encode(ts);  // {"nsec":123456789,"sec":1234567890}

// 反序列化：JSON 字符串 → C++ 结构体
auto result = foxglove::decode<Timestamp>(json_str);
```

### ADL to_json/from_json 模式

nlohmann::json 库通过 ADL（Argument Dependent Lookup）机制自动发现序列化函数。我们在 `foxglove` 命名空间内定义 `to_json` 和 `from_json` 函数，库会自动找到并调用它们：

```cpp
namespace foxglove {

// 序列化函数：结构体 → json 对象
void to_json(nlohmann::json& j, const Timestamp& ts) {
    j = nlohmann::json{{"nsec", ts.nsec}, {"sec", ts.sec}};
}

// 反序列化函数：json 对象 → 结构体
void from_json(const nlohmann::json& j, Timestamp& ts) {
    j.at("nsec").get_to(ts.nsec);
    j.at("sec").get_to(ts.sec);
}

}  // namespace foxglove
```

**为什么是 ADL 而非成员函数？**

1. **非侵入式**：可以在不修改类型定义的情况下添加序列化能力
2. **标准惯例**：nlohmann::json 推荐的方式
3. **灵活性**：可以支持第三方类型的序列化（通过包装）

### NaN → null 策略

Foxglove 协议要求浮点数的 NaN（Not a Number）值必须序列化为 JSON `null`：

```cpp
// 自定义 double 的序列化
inline void to_json(nlohmann::json& j, double value) {
    if (std::isnan(value)) {
        j = nullptr;  // NaN → null
    } else {
        j = value;
    }
}

// 自定义 double 的反序列化
inline void from_json(const nlohmann::json& j, double& value) {
    if (j.is_null()) {
        value = std::numeric_limits<double>::quiet_NaN();  // null → NaN
    } else {
        value = j.get<double>();
    }
}
```

**为什么要这样处理？**

- JSON 标准不支持 NaN/Infinity 字面值
- 如果直接序列化为字符串 "NaN"，接收方可能无法正确解析
- `null` 是 JSON 标准值，所有解析器都能正确处理
- 保证数据**可安全往返**：序列化 → 传输 → 反序列化 → 序列化，结果一致

### 字母序字段排序

Foxglove 协议要求 JSON 字段按字母顺序排列：

```cpp
// Timestamp 的正确顺序（nsec 在 sec 之前）
{"nsec":123456789,"sec":1234567890}

// Pose 的正确顺序（orientation 在 position 之前）
{"orientation":{"w":1.0,...},"position":{"x":1.0,...}}
```

nlohmann::json 默认使用 `std::map` 作为内部存储，天然保证按键的字母序排序。我们在构造 JSON 对象时只需按字母顺序添加字段即可。

**为什么要字母序？**

1. **确定性输出**：相同的输入总是产生相同的 JSON 字符串
2. **测试友好**：golden file 测试可以精确匹配
3. **协议规范**：Foxglove 官方要求

## Golden-File 测试方法论

Golden-file 测试是一种回归测试技术：将预期输出保存在文件中，测试时将实际输出与文件内容比较。

```cpp
TEST_CASE("Timestamp - golden file match") {
    Timestamp ts{1234567890, 123456789};
    auto encoded = encode(ts);
    
    // 读取预期输出
    std::string expected = read_file("tests/golden/timestamp.json");
    
    // 实际输出必须与 golden file 完全一致
    REQUIRE(encoded.value() == expected);
}
```

**Golden-file 的优点：**

- 精确验证输出格式（包括字段顺序、空格、数值精度）
- 易于更新：当格式变化时，只需更新 golden file
- 自文档化：golden file 本身就是期望输出的示例

**文件位置：**

```
tests/golden/
├── color.json        # Color 的期望输出
├── duration.json     # Duration 的期望输出
├── pose.json         # Pose 的期望输出
├── quaternion.json   # Quaternion 的期望输出
├── timestamp.json    # Timestamp 的期望输出
└── vector3.json      # Vector3 的期望输出
```

## 与官方 messages.hpp 的对比

| 特性 | 我们的实现 | 官方 SDK |
|------|-----------|---------|
| 序列化格式 | JSON（教学简化） | Protobuf（生产环境） |
| 模式发现 | ADL `to_json/from_json` | protobuf 生成代码 |
| 消息编码 | `encode<T>(msg)` 返回 JSON 字符串 | `msg.encode(ptr, len, &encoded_len)` |
| Schema 获取 | `T::json_schema()` 返回 JSON 对象 | `T::schema()` 返回 `Schema` 对象 |
| NaN 处理 | JSON `null` | protobuf 特定编码 |
| 字段顺序 | 字母序（JSON 要求） | 字段编号（protobuf 要求） |

**设计意图的差异：**

- 官方 SDK 追求性能和紧凑性，使用 protobuf
- 我们的实现追求可读性和教学价值，使用 JSON
- 两者在**语义层面**保持一致（相同的字段名、相同的默认值、相同的 NaN 处理逻辑）

## 默认值的约定

我们为每种类型定义了合理的默认值：

```cpp
struct Quaternion {
    double x = 0.0;
    double y = 0.0;
    double z = 0.0;
    double w = 1.0;  // 默认为单位四元数（无旋转）
};

struct Pose {
    Vector3 position;        // 默认 {0, 0, 0}
    Quaternion orientation;  // 默认 {0, 0, 0, 1}（无旋转）
};
```

**约定原则：**

- 数值类型默认为 0
- 四元数 w 默认为 1（单位四元数代表"无旋转"）
- Pose 默认表示原点处的无旋转姿态
- Color 默认表示全透明的黑色（rgba 全为 0）

## 实现细节

### 结构体定义

```cpp
struct Vector3 {
    double x = 0.0;
    double y = 0.0;
    double z = 0.0;
    
    static nlohmann::json json_schema();
};

// 相等比较（用于测试验证）
inline bool operator==(const Vector3& lhs, const Vector3& rhs) {
    return lhs.x == rhs.x && lhs.y == rhs.y && lhs.z == rhs.z;
}
```

### Schema 生成

```cpp
nlohmann::json Vector3::json_schema() {
    return {
        {"type", "object"},
        {"properties", {
            {"x", {{"type", "number"}}},
            {"y", {{"type", "number"}}},
            {"z", {{"type", "number"}}}
        }},
        {"required", {"x", "y", "z"}}
    };
}
```

### encode/decode 模板特化

```cpp
template <>
FoxgloveResult<std::string> encode<Vector3>(const Vector3& msg) {
    try {
        nlohmann::json j = msg;  // 调用 to_json
        return j.dump();          // 序列化为字符串
    } catch (const std::exception&) {
        return tl::make_unexpected(FoxgloveError::SerializationError);
    }
}

template <>
FoxgloveResult<Vector3> decode<Vector3>(const std::string& json_str) {
    try {
        nlohmann::json j = nlohmann::json::parse(json_str);
        return j.get<Vector3>();  // 调用 from_json
    } catch (const std::exception&) {
        return tl::make_unexpected(FoxgloveError::SerializationError);
    }
}
```

## 测试覆盖

本章测试涵盖以下场景：

1. **往返测试（Roundtrip）**：序列化 → 反序列化 → 验证相等
2. **Golden-file 匹配**：输出必须与预期完全一致
3. **NaN 处理**：NaN → null → NaN 的完整往返
4. **默认值验证**：所有字段的默认值正确
5. **Schema 验证**：返回的 JSON Schema 结构正确
6. **错误处理**：无效 JSON、缺少必需字段时返回错误

## 示例代码

见 `examples/ch05_serialization/main.cpp`：

```cpp
#include <foxglove/messages.hpp>
#include <cstdio>

using namespace foxglove;

int main() {
    // 创建并序列化一个 Pose
    Pose pose;
    pose.position = {1.0, 2.0, 3.0};
    pose.orientation = {0.0, 0.0, 0.0, 1.0};  // 单位四元数
    
    auto json = encode(pose);
    printf("Pose: %s\n", json.value().c_str());
    // 输出: {"orientation":{"w":1.0,"x":0.0,"y":0.0,"z":0.0},
    //         "position":{"x":1.0,"y":2.0,"z":3.0}}
    
    // 反序列化
    auto result = decode<Pose>(json.value());
    if (result.has_value()) {
        printf("Roundtrip OK\n");
    }
    
    return 0;
}
```
